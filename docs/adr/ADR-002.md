# ADR-002: Segurança de Segredos e Contexto Efêmero

**Status**: Aceito  
**Data**: 2024 (revisado 2026-06-16)

---

## Contexto

O projeto lida com credenciais de banco de dados, tokens de API, senhas SMTP e chaves de aplicação. Persistir qualquer desses valores em arquivos, logs ou no repositório git representa risco crítico de segurança (OWASP A02: Cryptographic Failures / A07: Identification and Authentication Failures).

---

## Decisão

### 1. Proibição total de hardcode

Nenhum segredo pode aparecer fixo em:
- `run.sh`
- `metadata.json`
- `CLAUDE.md` ou qualquer `.md` no repo
- Commits no git

**Verificação automática**: O script `run.sh` deve solicitar o segredo em tempo de execução via `read -s` ou receber via variável de ambiente do chamador.

### 2. Contexto Efêmero (Ephemeral Context)

O ciclo de vida de um segredo dentro do agente Claude é:

```
usuário digita segredo no chat
       ↓
Claude usa o valor na geração do YAML/comando
       ↓
stack é enviada ao Docker Swarm
       ↓
segredo NÃO é salvo em nenhum arquivo
       ↓
conversa encerrada → segredo descartado
```

### 3. O que salvar vs. não salvar

| Dado | Persistir? | Onde |
|------|-----------|------|
| URL de acesso | ✅ Sim | `/root/dados_vps/<skill>.md` |
| Nome de usuário (não-sensível) | ✅ Sim | `/root/dados_vps/<skill>.md` |
| Senha / token / chave | ❌ Nunca | — |
| App Key gerada dinamicamente | ❌ Nunca | Apenas na stack em memória |
| Referência ao YAML em disco | ✅ Sim | `/root/dados_vps/<skill>.md` |

### 4. Mascaramento em logs de chat

Quando o usuário fornecer um segredo, Claude deve:
- Confirmar recebimento sem repetir o valor: `"Senha SMTP recebida ✓"`
- Nunca exibir o valor em mensagens de status, resumos ou confirmações

### 5. Claude Code Skills e segredos

Se uma Claude Code Skill precisar de credenciais para operar (ex: validar conexão com banco), deve:
1. Solicitar via `AskUserQuestion` marcado como `sensitive`
2. Usar o valor apenas dentro do turno atual
3. Não incluir o valor em prompts injetados por `` !`cmd` `` (injeção dinâmica roda antes do chat)

### 6. Geração de segredos temporários

Segredos gerados em runtime (ex: `openssl rand -hex 32`) são aceitáveis se:
- Gerados dentro do `run.sh`, não commitados
- Passados diretamente ao `docker stack deploy` via heredoc/variável local
- Documentado no `README.md` da skill que a re-execução gera novo valor

---

## Consequências

**Prós**:
- Risco zero de vazamento via git history ou arquivos de log
- Compliance com OWASP Top 10 (A02, A07)
- Auditoria simplificada — segredos nunca em repouso em disco

**Contras**:
- Re-deploy requer re-fornecer credenciais (sem "lembrar" senhas)
- Usuário deve guardar senhas externamente (recomendado: Vaultwarden — ver `app-vaultwarden`)

---

## Alternativas Rejeitadas

- **`.env` file no repo**: risco direto de commit acidental
- **Variáveis de ambiente persistidas**: legíveis por qualquer processo com acesso ao host
- **Secrets do Docker Swarm**: viável tecnicamente, mas adiciona complexidade operacional não justificada para este perfil de uso
